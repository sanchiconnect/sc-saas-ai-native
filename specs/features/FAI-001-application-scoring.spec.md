# FAI-001: Application Scoring Flow

## Summary

The application scoring flow is the primary purpose of the `ai-startups-analyzer` service. A program manager in the admin panel uploads a CSV of startup applications; the analyzer batches them, scores each applicant using a configurable LLM provider (Gemini, OpenAI, or Anthropic), and returns structured ratings on a 1–5 scale. The admin panel is the sole intermediary: it triggers every step of the flow, polls for progress, finalizes the merged results, and then pushes the final scores back to `sc-saas-backend` — the analyzer never calls the backend directly. Optionally, an admin-drafted scoring thesis can be generated by the LLM in a preflight step before the CSV is uploaded.

## Flow sequence

1. **Admin: generate thesis (optional)**
   - Admin fills in program name, description, form fields, rating criteria, and application type in the admin panel.
   - Admin panel sends `POST /api/v1/generate-thesis/` with that metadata.
   - Analyzer builds a prompt and calls the active LLM provider via `run_chat_analysis_raw()`.
   - Analyzer returns `{thesis: string, cost: float}` — the USD cost of this call. The cost is not persisted.
   - Admin reviews and edits the thesis text in the UI before proceeding.

2. **Admin: upload CSV**
   - Admin sends `POST /api/v1/upload-csv/` with: applications CSV file, thesis text, `application_type`, `program_id`, `program_round_id`, and a `rating_criteria` JSON blob.
   - Analyzer validates the CSV, slices it into batches of `BATCH_SIZE` (default 5) rows.
   - Analyzer creates an `Analysis` DB row and one `Batch` DB row per slice; returns `run_id` (= `Analysis.id`, a UUID) plus batch count and applicant count metadata.
   - Individual batch CSV slices are written to disk at a path derived from `run_id`.

3. **Admin: start background analysis**
   - Admin sends `POST /api/v1/start-all-background/` with `run_id`.
   - Analyzer enqueues all batches as async background tasks (FastAPI `BackgroundTasks` or equivalent task queue).
   - Concurrency is controlled by two semaphores: `ANALYZER_GLOBAL_CONCURRENCY` (default 16) and `ANALYZER_PER_RUN_CONCURRENCY` (default 5).
   - Returns immediately; processing happens out-of-band.

4. **Batch processing (per batch)**
   a. Read the batch CSV slice from disk.
   b. Optionally run per-applicant enrichment — see FAI-002 for the enrichment sub-flow. Skipped when `ENABLE_ENRICHMENT=0`.
   c. Build scoring prompt: thesis + rating criteria + CSV row data + any enrichment evidence.
   d. Call `run_chat_analysis_raw()` against the provider configured in `DEFAULT_PROVIDER`.
   e. Parse LLM JSON response with `demjson3` (tolerant parser that handles trailing commas, single quotes, etc.).
   f. If `demjson3` fails: call `_salvage_objects()` to extract partial valid objects via regex/heuristics.
   g. Apply rating conversion — see **Scoring contract** below.
   h. Call `_merge_with_fallback()`: every applicant in the batch is guaranteed a result row; missing applicants receive `rating: 0` and a standard fallback justification string.
   i. Write `response.json` for this batch to disk. Update `responseStatus.json`. Update `Batch` DB row (`status`, `token_usage`, `cost_usd`).
   j. On failure after `MAX_BATCH_RETRIES` exhausted: mark batch as `error`; all applicants in the batch receive rating 0.

5. **Admin: poll progress**
   - Admin panel polls `GET /api/v1/status-summary/{run_id}` every few seconds via AJAX.
   - Response includes: `completed_batches`, `processing_batches`, `error_batches`, `total_batches`, `pct_complete` (0–100).
   - Admin panel renders a progress bar; no websocket — pure polling.

6. **Admin: finalize**
   - Admin sends `POST /api/v1/finalize-analysis/` with `run_id`.
   - Analyzer reads all per-batch `response.json` files, merges into one final JSON and one final CSV.
   - Rolls up total `token_usage` and `cost_usd` from all Batch rows onto the parent `Analysis` row.
   - Returns `{final_results_path: string, CostSummary: {...}}`.

7. **Scores stored in sc-saas-backend**
   - Admin panel reads the final JSON from the analyzer response.
   - Admin panel calls the `sc-saas-backend` REST API to persist a `rating` (decimal) per application record.
   - The backend column is `decimal(4,3)`; scores arrive as floats already in the 1–5 range.
   - The analyzer has no network access to `sc-saas-backend` and does not call it directly.

## Scoring contract

The analyzer works on a **0–500 raw model scale** and always delivers a **1–5 decimal** to the admin panel.

**Holistic mode** (no rating criteria provided):
- LLM returns a single integer `0–500` per applicant.
- `_coerce_rating(raw)`: `rating = clamp(raw / 100, 1.0, 5.0)` rounded to 2 decimal places.
- A raw score of 0 from a fallback (model did not return one) yields `rating: 0` — a sentinel value, not a real score.

**Criteria mode** (rating criteria JSON provided):
- `rating_criteria` JSON shape: `{ratingMethod: string, criterias: [{name: string, weightage: number}]}`.
- LLM returns a per-criterion 0–500 score for each criterion name.
- `_weighted_rating(scores, criterias)`: weighted sum of `(score * weightage)` across criteria, then `÷ 100`, clamped to 1–5.
- Weights must sum to 100 (validated in the prompt construction step; not enforced in `_weighted_rating` itself).

**Landing in sc-saas-backend:**
- Admin writes the 1–5 float to the backend via REST. The backend `decimal(4,3)` column stores e.g. `4.25`.
- NestJS TypeORM entity uses a numeric transformer because MySQL returns decimals as strings; the transformer calls `parseFloat()` before returning to callers.

## Configuration

| Variable | Default | Effect |
|---|---|---|
| `DEFAULT_PROVIDER` | `gemini` | Selects the LLM backend. Options: `gemini`, `openai`, `anthropic`. |
| `BATCH_SIZE` | `5` | Applicants per LLM call. Larger values reduce call count but increase per-call token load and timeout risk. |
| `MAX_BATCH_RETRIES` | `3` | Retry attempts on rate-limit (429) or timeout before marking the batch as error. |
| `BATCH_TIMEOUT_S` | `600` | Hard ceiling in seconds per batch LLM call. Enforced via `asyncio.wait_for`. |
| `ANALYZER_GLOBAL_CONCURRENCY` | `16` | Semaphore cap on total in-flight LLM calls across all active runs and tenants. |
| `ANALYZER_PER_RUN_CONCURRENCY` | `5` | Semaphore cap on concurrent batches within a single run. |
| `CLOUD_API_KEY` | (required) | LLM provider API key. Single value shared across all tenants. |
| `ENABLE_ENRICHMENT` | `0` | Master switch for pre-scoring applicant enrichment (see FAI-002). |

## Auth & access

- All admin → analyzer calls use **Bearer token authentication**.
- The token is validated by the analyzer via `bcrypt.checkpw()` against hashed keys stored in the `api_keys` table.
- The `matched_key.domain` field is extracted from the validated key record and used downstream in `pricing.py` to apply per-domain cost overrides (different tenants may have different billing rates).
- No per-endpoint ACL beyond the API key check — any valid key can call any endpoint.
- The admin panel stores its analyzer API key in its `.env` / `config.php`; it is never sent to `sc-saas-backend` or `sc-saas-frontend`.

## Error handling

| Failure mode | Behavior |
|---|---|
| LLM rate limit (429) | Retry with exponential backoff, up to `MAX_BATCH_RETRIES`. If exhausted, batch status → `error`. |
| LLM timeout (`> BATCH_TIMEOUT_S`) | `asyncio.TimeoutError` caught; same retry path. |
| JSON parse failure (`demjson3` raises) | `_salvage_objects()` attempts regex extraction of partial objects. If salvage yields ≥ 1 object, scoring continues with partial results; unmatched applicants get rating 0 via `_merge_with_fallback()`. |
| Complete batch failure (all retries + salvage fail) | All applicants in the batch receive `rating: 0` and a standard fallback justification. Batch row status → `error`. |
| Finalize called before all batches complete | Finalize merges whatever `response.json` files exist. Batches still in `processing` state produce no file; those applicants silently receive no row in the final output — no error is raised. |
| Disk write failure (response.json) | Unhandled — will surface as a 500 from the batch task. Dual-status drift risk (see Known issues). |

## Known issues

1. **Shared `CLOUD_API_KEY` across all tenants.** A single API key is used for every LLM call regardless of which tenant triggered the run. If one tenant's large analysis triggers a provider-side rate limit on that key, all other simultaneously running analyses are also rate-limited. There is no per-tenant key isolation or quota enforcement at the analyzer level.

2. **Dual status tracking can drift on crash.** Each batch writes results in two places: the `Batch` DB row and `responseStatus.json` on disk. These are updated in sequence — if the process crashes between the DB write and the file write (or vice versa), the two sources of truth become inconsistent. `status-summary` reads from the DB; `finalize-analysis` reads from the files. A batch that appears `completed` in the DB but has no `response.json` on disk will silently produce missing rows in the final output.

3. **`routes_back.py` ghost file.** A 2,434-line legacy backup (`routes_back.py`) lives in the same directory as the active `routes.py`. It is not imported anywhere, but it creates a real risk of confusion about which file governs production behavior. Any developer editing what they believe is the active routes file may be editing the dead copy.

4. **No test suite on scoring logic.** `_coerce_rating()`, `_weighted_rating()`, `_salvage_objects()`, and `_merge_with_fallback()` have no automated tests. Changes to any of these functions (e.g. clamping bounds, weight normalization) cannot be validated before deploy and carry direct correctness risk for stored application scores.

5. **Worst-case serial runtime.** With `BATCH_SIZE=5`, a 100-applicant CSV produces 20 batches. At `BATCH_TIMEOUT_S=600` per batch and `ANALYZER_PER_RUN_CONCURRENCY=5`, the worst case (all batches time out) is 2,400 seconds (~40 minutes) of retried work per run. A single large tenant run (e.g. 200 applicants) can hold all 16 global concurrency slots for an extended period and starve other tenants.

6. **Finalize is not idempotent with error batches.** Calling `POST /api/v1/finalize-analysis/` when some batches are in `error` state produces a final CSV that silently omits those applicants. There is no warning in the `CostSummary` response or the admin UI that the output is incomplete.
